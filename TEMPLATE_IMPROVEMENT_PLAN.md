**目标**: 打造能基于 tasks 任务自主顺利运转的 Agent 团队

## 📋 改进计划总览

基于用户会话分析，发现 14 次用户指导/纠正，主要集中在：
1. 任务完成标准错误（任务完成 ≠ 项目完成）
2. PM 不理解角色 ID 映射
3. Agent 轻易放弃（说"做不到"）
4. PM 自己做操作而非分配任务

### 1. TEAMWORK_FLOW.md 改进方案

**文件位置**: `templates/standard-dev/TEAMWORK_FLOW.md`

**改进内容**:

```markdown
# 团队工作流程 (TEAMWORK_FLOW)

## 任务流水线

```
需求分析(产品) → 技术设计(可跳过) → 开发(前端/后端) → 测试(测试) → 验收(项目经理)
```

## 阶段定义

| 阶段 | 默认执行人 | 产出要求 |
|------|-----------|---------|
| 需求分析 | 产品 | PRD/需求文档，明确验收标准，功能点列表 |
| 技术设计 | 前端研发/后端研发 | 技术方案文档，接口设计，数据库设计 |
| 开发 | 前端研发/后端研发 | 代码提交完毕，**PR 已合并到主分支** |
| 测试 | 测试 | 测试报告，Bug 列表（若有），回归完成 |
| 验收 | 项目经理 | 验收结论，上线确认 |

## ⚠️ 核心原则：任务完成定义

### 开发任务完成 = PR 已合并

**重要**：任务"进展"变为"已完成" ≠ 项目完成

| 任务类型 | 完成标准 |
|---------|---------|
| 开发任务 | PR 已合并到主分支 |
| 测试任务 | 测试报告已输出 |
| 验收任务 | 用户确认验收通过 |
| 需求任务 | 需求文档已评审 |

### PM 核心职责：分配 vs 执行

**你是 PM，不是执行者。**

| 应该做的 | 不应该做的 |
|----------|------------|
| 分配任务给对应 Agent | 自己调用 API 执行操作 |
| 协调各方资源 | 自己合并 PR |
| 跟踪进度 | 自己解决代码冲突 |
| 识别阻塞并推动解决 | 自己执行应该由 Worker 执行的操作 |

**例外**：当你被明确授权执行某操作时，才可以自己执行。

## 流转规则

### 正常流转

1. 任务"进展"变为"已完成"后，主 Agent 负责流转
2. 流转时：
   - 将"进展"重置为"待开始"
   - 更新"任务执行人"为下一阶段负责人
   - 更新"任务阶段"
   - 可在"最新进展记录"中添加流转备注
3. 主 Agent 可根据任务复杂度跳过阶段（如小需求跳过"技术设计"）

### ⚠️ 流转前必须检查 PR 状态

对于"开发"阶段任务：
- 任务详情中必须有 PR 链接
- 只有 PR 已合并，才可以流转到下一阶段
- 如果 PR 未合并，先分配 Code Review/合并任务

### 回退规则

- 测试阶段发现 Bug → 打回"开发"阶段
- 验收不通过 → 根据具体情况打回相应阶段
- PR 有冲突 → 打回"开发"阶段（分配给原开发者）

### 流转示例

```bash
# 需求分析完成，流转到开发
openclaw-team task flow recXXX --next-assignee "前端研发" --stage "开发"

# 测试完成，流转到验收
openclaw-team task flow recXXX --next-assignee "项目经理" --stage "验收"

# 开发任务，但 PR 未合并 → 先分配合并任务
# （不要直接流转，等待 PR 合并后再流转）

# 验收通过，标记完成
openclaw-team task flow recXXX --next-assignee "项目经理" --stage "完成"
```

---

## 主 Agent 职责

- 每 10 分钟检查一次任务表
- 检查"已完成"任务 → **必须先检查 PR 状态** → 按流水线流转
- 检查"进行中"超 24 小时任务 → 评估是否需要介入
- 检查 PR 状态 → 发现未合并的 PR → 分配合并/Code Review 任务
- 状态变化时在飞书群同步简报

---

## 禁止行为（红线）

1. **禁止说"我做不到"** - 遇到问题先尝试 3 种以上方案
2. **禁止跳过分配直接执行** - 需要操作时，先分配给对应角色
3. **禁止忽略 PR 状态** - 任务完成必须等 PR 合并
4. **禁止使用错误的角色 ID** - 必须使用映射表中的 ID

---

## 任务优先级

- **P0**: 紧急，需立即处理，阻塞其他任务
- **P1**: 高优先级，当日完成
- **P2**: 正常优先级（默认）
- **P3**: 低优先级，可推迟
```

### 2. pm-check.md 改进方案

**文件位置**: `templates/standard-dev/cron-prompts/pm-check.md`

**改进内容**:

```markdown
你是项目经理，正在执行定时任务检查。快速完成以下步骤，有变化时汇报，无变化则静默结束。

## 执行步骤

**第 1 步：获取任务全景**

```bash
openclaw-team task list --format json
```

**第 2 步：处理已完成任务 → 流转**

对于每个"进展"为"已完成"的任务：
1. 阅读"任务情况总结"，评估交付质量
2. **必须先检查 PR 状态**（见下方）
3. 对照 TEAMWORK_FLOW.md 确定下一阶段和执行人
4. 流转任务：
   ```bash
   openclaw-team task flow <record_id> --next-assignee "下一执行人" --stage "下一阶段"
   ```
5. 若处于"验收"阶段且验收通过，设为"完成"

**第 2.5 步：检查 PR 状态（重要！⚠️）**

对于"开发"阶段且"已完成"的任务：
1. 从任务详情中提取 PR 链接
2. 用以下命令检查 PR 状态：
   ```bash
   gh pr view <PR号> --json state,mergeable,reviews
   # 或
   gh pr status
   ```
3. 根据 PR 状态决定下一步：

| PR 状态 | 操作 |
|---------|------|
| 已合并 | ✅ 正常流转到下一阶段 |
| pending review | 🔄 分配 Code Review 任务给其他 Agent |
| has conflicts | ⚠️ 分配解决冲突任务给原开发者 |
| approved, 未合并 | ⚠️ 分配合并任务给其他 Agent |
| 未创建 PR | ⚠️ 分配创建 PR 任务给原开发者 |

4. **只有 PR 已合并，才可以流转任务**

**第 3 步：检查异常任务**

- "进行中"超过预期时间的任务 → 在进展记录中添加备注询问进展
- 存在阻塞标记的任务 → 评估是否需要重新分配或调整方案
- 发现有"已完成"但 PR 未合并的任务 → **这是异常！** 需要处理

**第 4 步：反馈汇报**

若有任务状态发生变化，在飞书群发送简洁摘要：

```
📊 任务进展更新
━━━━━━━━━━━━━━
✅ 已流转：[任务名] → [下一阶段]([执行人])
🔄 PR 状态：[N个 PR 待合并]
⚠️ 需关注：[问题描述]（如有）

若无任何变化，静默结束，不发送消息。
```

## ⚠️ 常见问题处理

### Q: 任务标记"已完成"但 PR 未合并？

A: 这是数据不一致。需要：
1. 将任务状态改回"进行中"
2. 分配合并/解决冲突任务
3. 等 PR 合并后再重新标记完成

### Q: 如何识别需要合并的 PR？

A: 查找同时满足以下条件的任务：
- 阶段 = "开发"
- 进展 = "已完成"（或最近更新超过 2 小时）
- 任务详情中有 PR 链接

### Q: PR 检查失败怎么办？

A:
1. 如果是权限问题，先分配任务让对应 Agent 检查
2. 如果是 Token 问题，记录下来报告给用户
3. 不要跳过 PR 检查直接流转任务
```

  ↓
