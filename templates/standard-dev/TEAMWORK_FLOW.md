# 团队工作流程 (TEAMWORK_FLOW)

## 任务流水线

```
需求分析(产品) → 技术设计(可跳过) → 开发(前端/后端) → 测试(测试) → 验收(项目经理)
```

## 阶段定义

| 阶段 | 默认执行人 | 产出要求 |
|------|-----------|---------|
| 需求分析 | 产品 | PRD/需求文档，明确验收标准，功能点列表 |
| 技术设计 | 前端研发/后端研发 | 技术方案文档，接口设计，数据库设计 |
| 开发 | 前端研发/后端研发 | **PR 已合并到主分支** |
| 测试 | 测试 | 测试报告，Bug 列表（若有），回归完成 |
| 验收 | 项目经理 | 验收结论，上线确认 |

## ⚠️ 核心原则：任务完成定义

### 开发任务完成 = PR 已合并

**重要**：任务"进展"变为"已完成" ≠ 项目完成

| 任务类型 | 完成标准 |
|---------|---------|
| 开发任务 | **PR 已合并到主分支** |
| 测试任务 | 测试报告已输出 |
| 验收任务 | 用户确认验收通过 |
| 需求任务 | 需求文档已评审 |

### PM 核心职责：分配 vs 执行

**你是 PM，不是执行者。**

| 应该做的 | 不应该做的 |
|----------|------------|
| 分配任务给对应 Agent | 自己调用 API 执行操作 |
| 协调各方资源 | 自己合并 PR |
| 跟踪进度 | 自己解决代码冲突 |
| 识别阻塞并推动解决 | 自己执行应该由 Worker 执行的操作 |

**例外**：当你被明确授权执行某操作时，才可以自己执行。

## 角色ID映射表

**重要**：在任务管理表中填写执行人时，必须使用以下 ID：

| 角色 | 可用 ID | 实际 Agent | 说明 |
|------|---------|------------|------|
| 产品 | `product`, `产品` | product | 产品经理，负责需求分析 |
| 前端 | `fe`, `前端`, `frontend` | fe | 前端开发工程师 |
| 后端 | `be`, `后端`, `backend` | be | 后端开发工程师 |
| 测试 | `qa`, `测试`, `test` | qa | 测试工程师 |
| 项目经理 | `pm`, `项目经理` | pm | 项目总控，负责任务分配和流转 |
| 运维 | `ops`, `运维` | ops | 运维工程师，负责部署和发布 |

**禁止**使用 Agent 的实际名称（如 Beta, Alpha, 大虾）

### 使用示例

```bash
# ✅ 正确
openclaw-team task create --title "实现登录" --assignee "fe"
openclaw-team task flow recXXX --next-assignee "be" --stage "开发"

# ❌ 错误
openclaw-team task create --title "实现登录" --assignee "前端"
openclaw-team task create --title "实现登录" --assignee "Beta"
```

## 流转规则

### 正常流转

1. 任务"进展"变为"已完成"后，主 Agent 负责流转
2. 流转时：
   - 将"进展"重置为"待开始"
   - 更新"任务执行人"为下一阶段负责人
   - 更新"任务阶段"
   - 可在"最新进展记录"中添加流转备注
3. 主 Agent 可根据任务复杂度跳过阶段（如小需求跳过"技术设计"）

### ⚠️ 流转前必须检查 PR 状态

对于"开发"阶段任务：
- 任务详情中必须有 PR 链接
- **只有 PR 已合并，才可以流转到下一阶段**
- 如果 PR 未合并，先分配 Code Review/合并任务

### 回退规则

- 测试阶段发现 Bug → 打回"开发"阶段
- 验收不通过 → 根据具体情况打回相应阶段
- PR 有冲突 → 打回"开发"阶段（分配给原开发者）

### 流转示例

```bash
# 需求分析完成，流转到开发
openclaw-team task flow recXXX --next-assignee "fe" --stage "开发"

# 测试完成，流转到验收
openclaw-team task flow recXXX --next-assignee "pm" --stage "验收"

# 开发任务，但 PR 未合并 → 先分配合并任务
# （不要直接流转，等待 PR 合并后再流转）

# 验收通过，标记完成
openclaw-team task flow recXXX --next-assignee "pm" --stage "完成"
```

## 禁止行为（红线）

1. **禁止说"我做不到"** - 遇到问题先尝试 3 种以上方案
2. **禁止跳过分配直接执行** - 需要操作时，先分配给对应角色
3. **禁止忽略 PR 状态** - 任务完成必须等 PR 合并
4. **禁止使用错误的角色 ID** - 必须使用映射表中的 ID

---

## 主 Agent 职责

- 每 10 分钟检查一次任务表
- 检查"已完成"任务 → **必须先检查 PR 状态** → 按流水线流转
- 检查"进行中"超 24 小时任务 → 评估是否需要介入
- 检查 PR 状态 → 发现未合并的 PR → 分配合并/Code Review 任务
- 状态变化时在飞书群同步简报

---

## 任务优先级

- **P0**: 紧急，需立即处理，阻塞其他任务
- **P1**: 高优先级，当日完成
- **P2**: 正常优先级（默认）
- **P3**: 低优先级，可推迟
